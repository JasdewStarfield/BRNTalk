# BRNTalk Roadmap

当前版本基线：`1.0.2`

## 定位

- 面向 RPG 服务器、任务地图和整合包的剧情对话模组
- 以服务端驱动、多人同步、热重载为核心
- 长期目标是成为可复用的剧情对话基础设施，而不只是一个聊天 UI

## 当前判断

- 现阶段基础功能已经成型
- 下一步重点应放在稳定性、作者体验和扩展能力
- UI 体验仍重要，但不建议排在最前面

## 路线原则

1. 先补稳定性，再补大功能。
2. 先补作者工作流，再补复杂表现层。
3. 优先做可复用的平台能力，少做一次性特化逻辑。
4. 服务端行为开关优先走 `serverconfig`。

## Phase 1：1.0.x 稳定化

### 目标

- 降低回归风险
- 固化发布流程
- 提升脚本排错效率
- 这一阶段只更新仓库，不涉及模组版本发布

### 任务

- 补第一批自动化测试
  - 断链
  - 重复 ID
  - 空 choice
  - 无限自动推进
  - `wait` 恢复
  - `action` 执行
- 固化构建与发布流程
  - 默认使用 `.\gradlew.bat build --no-configuration-cache`
- 整理 `validation_fixtures`
- 补一份脚本排错说明
- 明确客户端大类的后续拆分边界

## Phase 2：1.1 作者体验与联动

### 目标

- 更容易写剧本
- 更容易接入其他系统
- 更清晰地暴露公共能力

### 任务

- 完善脚本文档和推荐写法
- 增强校验输出
  - 尽量定位到文件、剧本 ID、消息 ID
- 梳理公开 API 的稳定边界
- 增加更合适的联动事件点
- 评估对 KubeJS、任务系统、命令编排的支持方式
- 预留更正式动作系统的设计空间

## Phase 3：1.2 客户端结构与体验

### 目标

- 改善长对话和多线程体验
- 降低 UI 维护成本

### 任务

- 拆分 `TalkScreen`
- 拆分 `ClientTalkUtils`
- 优化线程列表与未读状态
- 优化长对话预览
- 继续打磨 HUD / Toast 行为一致性
- 视情况评估以下能力
  - 搜索消息
  - 重要消息置顶
  - 对话分组

## Phase 4：平台化增强

### 候选方向

- 条件节点
- 变量与标签系统
- 更正式的动作系统
- 更丰富的消息载体
- 更稳定的跨模组集成协议

## 独立探索线

### Forge `1.20.1` / 旧平台兼容

- 建议单独评估
- 不建议混入主线近期版本
- 必要时单独开实验分支

## 暂不优先

- 大规模视觉重做
- 过早引入复杂剧情 DSL
- 在缺少测试前提下做大范围重构
- 为了堆功能而加入低复用特殊节点
- 主线未稳时同步推进跨平台迁移

## 近期节奏建议

- `1.0.3`
  - 测试、构建流程、脚本排错文档
- `1.1.0`
  - 作者体验、校验输出、联动能力
- `1.2.0`
  - 客户端结构拆分、UI 体验增强
- `2.x`
  - 视实际使用情况再决定是否推进条件/变量/动作系统升级
